# Pi optimization notes

Two design decisions span several files and are easy to break if you don't know
they exist. Everything else is explained in inline comments where it lives.

## 1. The playout always wins the CPU

A DeckPilot box is a playout deck first. mpv decode/render must never starve,
even while the operator dumps 50 clips onto the SMB share mid-show. The priority
gradient is built from three independent layers:

- **The whole service runs at `Nice=-5`** (`deploy/pideck-open.service` and the
  unit generated by `scripts/bootstrap.sh`). The Python app spawns `cage`, cage
  spawns mpv, and both inherit it — so the entire playback chain sits above
  anything else on the Pi.
- **Background ffmpeg work renices itself `+19` *relative*** via `os.nice(19)` in
  `_background_run()` (`app/media/clip_store.py`), landing at ~14 — well below
  playback. Because `os.nice` is a *relative* increment, this composes correctly
  with the service `Nice` and needs no absolute value. On Linux it is also
  wrapped in `ionice -c3` (idle I/O class) so a full-file re-encode can't stall
  SD-card reads from the clip on air.
- **ffmpeg thread caps** (`-threads 2` on conform, `-threads 1` on audio levels)
  stop a single import from grabbing all four cores.

If you add a new ffmpeg/ffprobe call, route it through `_background_run()` — not
`subprocess.run` — or it will fight playback.

## 2. Progressive ingestion: the `settle_paths` contract

When files are dropped over SMB/USB, half-copied files must never enter the
library (a probe of a partial file gives garbage), and a big drop must be
ingested **progressively** — each file the moment *it* finishes copying, not
after the whole batch goes quiet.

The watch folder owns "is this file done copying?"; the clip store owns "insert
and enqueue". They meet through one optional argument:

```
WatchFolderService.tick()                 # per-file stability: size+mtime equal
  └─ DeckController.refresh_clips(settle_paths)   #   across two consecutive scans
       └─ ClipStore.sync_with_disk(settle_paths)  # ingest ONLY settled paths
```

- `settle_paths` is the set of absolute paths that have stopped changing.
- The watch folder passes it on every tick, so still-copying files are skipped
  and picked up on a later pass.
- **Every other caller** of `refresh_clips` / `sync_with_disk` (operator actions,
  startup) passes `None`, which means "ingest everything on disk" — the original
  behavior. The gate is opt-in and only the watch folder uses it.

Removals are also two-tick (a file must be absent from two consecutive scans) so
a USB brown-out is never read as a mass delete.
